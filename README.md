# График дежурств

[![Тесты](https://github.com/SimonKabb/work_schedule/actions/workflows/tests.yml/badge.svg?branch=main)](https://github.com/SimonKabb/work_schedule/actions/workflows/tests.yml)

Веб-приложение для подготовки графиков дежурств нескольких медицинских коллективов. Система учитывает нормы часов, тип занятости, пожелания сотрудников и правила распределения смен.

## Возможности

- несколько изолированных коллективов в одной системе;
- главный администратор и заведующие с ограничением доступа по коллективам;
- регистрация сотрудников по персональной ссылке коллектива или QR-коду;
- отзыв старой ссылки регистрации и выпуск новой;
- основные сотрудники и совместители с разными нормами часов;
- отдельные типы смен для будней и выходных;
- календарь пожеланий «Могу» и «Не могу»;
- годовой календарь отпусков с выбором отдельных дней и диапазонов;
- отображение отпусков в месячном графике и запрет назначения смен в отпуск;
- автоматическая генерация графика с помощью OR-Tools;
- защита от смен подряд и распределение нагрузки;
- просмотр графика по месяцам;
- экспорт выбранного месяца в Excel;
- административная панель Django.

## Роли

| Роль | Возможности |
|---|---|
| Главный администратор | Создаёт коллективы и пользователей, назначает заведующих, управляет всеми данными |
| Заведующий | Управляет только своими коллективами, сменами, графиками, сотрудниками и приглашениями |
| Сотрудник | Видит график своих коллективов и отмечает пожелания по датам |

Заведующий может участвовать в графике как сотрудник или только управлять коллективом.

## Регистрация сотрудников

Для каждого коллектива создаётся уникальное приглашение:

1. Заведующий открывает график нужного коллектива.
2. Нажимает «Регистрация сотрудников».
3. Копирует ссылку или показывает QR-код сотруднику.
4. Сотрудник создаёт логин и пароль.
5. Система автоматически добавляет его только в предназначенный коллектив.

Если приглашение попало посторонним, заведующий может заменить ссылку и QR-код. Предыдущее приглашение сразу перестанет работать.

## Локальный запуск

Требуется Python 3.11.

```bash
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
python core/manage.py migrate
python core/manage.py createsuperuser
python core/manage.py runserver
```

После запуска:

- график: [http://127.0.0.1:8000/schedule/](http://127.0.0.1:8000/schedule/);
- административная панель: [http://127.0.0.1:8000/admin/](http://127.0.0.1:8000/admin/).

По умолчанию используется локальная база `core/db.sqlite3`.

## Настройки окружения

| Переменная | Назначение | Значение по умолчанию |
|---|---|---|
| `DJANGO_SECRET_KEY` | Секретный ключ Django | Локальный ключ для разработки |
| `DJANGO_DEBUG` | Режим отладки | `true` |
| `DJANGO_ALLOWED_HOSTS` | Разрешённые хосты через запятую | `localhost,127.0.0.1` |
| `DJANGO_CSRF_TRUSTED_ORIGINS` | Доверенные HTTPS-адреса | Пусто |
| `DJANGO_DB_PATH` | Путь к SQLite-базе | `core/db.sqlite3` |
| `DJANGO_STATIC_ROOT` | Каталог собранной статики | `core/staticfiles` |

Для рабочего сервера обязательно задайте собственный `DJANGO_SECRET_KEY`, отключите `DJANGO_DEBUG` и укажите домен в `DJANGO_ALLOWED_HOSTS` и `DJANGO_CSRF_TRUSTED_ORIGINS`.

## Тесты

Локальная проверка:

```bash
python core/manage.py check
python core/manage.py makemigrations --check --dry-run
python core/manage.py test work_schedule
```

GitHub Actions выполняет эти проверки при каждом push в `main` и для каждого pull request. Плашка в начале README показывает результат последнего запуска в основной ветке. Настройка находится в [`.github/workflows/tests.yml`](.github/workflows/tests.yml).

## Docker

Сборка и запуск контейнера приложения:

```bash
docker build -t work-schedule .
mkdir -p data
docker run --rm \
  -e DJANGO_DB_PATH=/app/data/db.sqlite3 \
  -v "$(pwd)/data:/app/data" \
  work-schedule python manage.py migrate
docker run --rm -p 8000:8000 \
  -e DJANGO_DEBUG=true \
  -e DJANGO_DB_PATH=/app/data/db.sqlite3 \
  -v "$(pwd)/data:/app/data" \
  work-schedule
```

Перед рабочим запуском также необходимо собрать статические файлы и задать безопасные переменные окружения либо подключить внешнюю базу данных.
